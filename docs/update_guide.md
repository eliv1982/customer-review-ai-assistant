# Инструкция по обновлению ассистента

---

## 1. Как обновлять системный промпт

1. Откройте файл **`prompts.py`**.
2. Отредактируйте константу **`SYSTEM_PROMPT`** (и при необходимости **`ASSISTANT_NAME`**).
3. Сохраните правки в кодировке **UTF-8**.
4. **Не удаляйте** из промпта явные требования к JSON-полям (`sentiment`, `topic`, `summary`, `reply_draft`), если не меняете одновременно схему в **`services/ai_service.py`**.
5. После изменений выполните ручную проверку (раздел 5) или включите smoke (раздел 7).

Рекомендуется дублировать смысловые изменения в **`docs/assistant_prompt_for_docs.md`** и при правках сценариев — в **`docs/scenarios_for_docs.md`**.

---

## 2. Как обновлять `knowledge_base.csv`

1. Откройте **`data/knowledge_base.csv`** в редакторе с поддержкой UTF-8 (и при необходимости UTF-8 с BOM).
2. Сохраняйте **заголовок** первой строкой; имена колонок должны совпадать с ожидаемыми в коде (`services/knowledge_base_service.py`, константа `REQUIRED_COLUMNS`).
3. Не оставляйте пустую строку `topic` у записей — такие строки при загрузке могут быть пропущены с предупреждением в логе.

После сохранения перезапуск бота достаточен: файл читается при обработке отзыва.

---

## 3. Как добавлять новые шаблоны

1. Добавьте **новую строку** в `knowledge_base.csv`.
2. Заполните поля:
   - **`review_type`**: `complaint` | `gratitude` | `suggestion` | `mixed`;
   - **`common_phrase`**: типовая формулировка клиента;
   - **`sentiment`** и **`topic`**: те же литералы, что ожидает модель в JSON;
   - **`reply_template`**: пример ответа для оператора;
   - **`recommended_action`**, **`summary_example`**: по желанию, для внутреннего использования.
3. Убедитесь, что **тональность строки** не конфликтует с очевидным смыслом фразы (иначе подбор может отдать менее релевантный шаблон).

Шаблоны **не подменяют** `reply_draft` модели; они показываются как **справка** в Telegram (они показываются как справочная подсказка в Telegram после отбора наиболее подходящих совпадений).

---

## 4. Переменные окружения

1. Файл **`.env`** (шаблон — **`.env.example`**).
2. **`OPENAI_MODEL`** — модель чата (например, `gpt-4o-mini`).
3. **`DATABASE_PATH`** — путь к файлу SQLite.
4. **`TELEGRAM_BOT_TOKEN`**, **`OPENAI_API_KEY`** — обязательны для полного цикла в боте.

---

## 5. Как проверять проект после изменений

1. **Зависимости:** `pip install -r requirements.txt` (в активированном venv).
2. **Конфигурация:** файл **`.env`** из **`.env.example`**, заполнены `OPENAI_API_KEY`, при проверке бота — `TELEGRAM_BOT_TOKEN`.
3. **Запуск:** `python main.py` — инициализируется БД, поднимается бот (если задан токен).
4. **Ручная проверка в Telegram:**
   - `/start`, `/help`, `/new_review`;
   - отправить **короткий тестовый** отзыв без персональных данных;
   - убедиться, что приходят тональность, тема, сводка, черновик и блок шаблонов (если есть подходящие строки в KB);
   - повторить **тот же текст** в течение 24 часов — должна сработать **защита от дублей** (сообщение о ранее обработанном отзыве).
5. **`/report`** — убедиться, что отчёт формируется без ошибки.

---

## 6. Обычный режим запуска

Из корня репозитория:

```bash
python main.py
```

- Выполняется **`init_database`**: таблицы, индексы, при необходимости backfill текстовых полей отзыва.
- **Smoke-тесты не запускаются** по умолчанию.
- При наличии **`TELEGRAM_BOT_TOKEN`** стартует **long polling** (aiogram).

Остановка: **Ctrl+C** в терминале.

---

## 7. Как включать smoke-тесты только при необходимости

Smoke предназначены для **локальной** проверки цепочки: демо-импорт CSV, отчёт по полной базе, загрузка KB, один прогон `process_review` с тестовым источником (при наличии ключа OpenAI).

**Windows (cmd):**

```bat
set RUN_SMOKE_TESTS=true
python main.py
```

**Windows (PowerShell):**

```powershell
$env:RUN_SMOKE_TESTS="true"; python main.py
```

Значения переменной, которые считаются «включено»: `1`, `true`, `yes`, `on` (без учёта регистра), см. **`main.py`**.
