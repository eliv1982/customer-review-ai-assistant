# Примеры аналитики и структура отчёта

Документ описывает основные показатели аналитики по отзывам, используемые в проекте, и поясняет, как интерпретировать результаты отчёта.

---

## Откуда берутся данные

- Таблица **`reviews`**: источник, текст, статус, оценка (`rating`, может быть `NULL`), товар, клиент, время создания.
- Таблица **`review_analysis`**: тональность, тема, сводка, черновик (после успешного вызова OpenAI).
- Отчёт **`/report` в Telegram** строится на **отфильтрованном** снимке: не учитываются источники с префиксом `smoke_`, в блоке по товарам скрыты пустые названия и заглушка «Не указано» (см. `README.md`).

---

## Показатели и связь с данными (фактическая реализация)

| Показатель | Источник в SQLite | Примечание |
|------------|-------------------|------------|
| Число отзывов, распределение по статусу | Таблица **`reviews`** (`status`, `created_at`) | Статусы: `new`, `processed`, `error`. |
| Распределение по тональности и темам | Таблица **`review_analysis`** (JOIN с `reviews` при фильтрации по источнику) | Литералы `sentiment` и `topic` из ответа модели. |
| Средняя оценка | **`reviews.rating`** | `AVG(rating)` только по строкам, где `rating IS NOT NULL`. |
| Сводка по товарам | **`reviews.product_name`**, **`rating`** | Группировка по названию; в Telegram-отчёте — усечённый список и фильтры «без названия» / «Не указано». |
| Доля каналов (Telegram / CSV и др.) | Поле **`reviews.source`** | В текущем MVP отдельной диаграммы в боте нет, но данные можно агрегировать SQL-запросами по префиксу или значению `source`. |

Функция сборки снимка: **`build_analytics_snapshot`** в `services/report_service.py`.

## Расширение отчёта (идеи на будущее)

Для портфолио или следующей итерации продукта к текстовому отчёту можно добавить:

- **Период** — явные даты «с» / «по» по `reviews.created_at` (сейчас отчёт по всей выборке без среза по датам в UI).
- **Краткое текстовое резюме** в свободной форме (1–2 абзаца) — вручную аналитиком или отдельным вызовом ИИ по уже посчитанным цифрам.
- **Топ-N тем** с количеством (частично уже есть в блоке тем и «сложных» тем).
- **Оговорка об ограничениях** — мало записей за период, перекос источников, неполные оценки.

## Пример 1: аналитика по темам

**Фрагмент смысла (не дословный вывод программы):**

| Тема (в БД / JSON) | Человекочитаемо | Кол-во отзывов с анализом |
|--------------------|-----------------|---------------------------|
| `delivery` | доставка | 4 |
| `product_quality` | качество товара | 3 |
| `other` | прочее | 2 |

**Интерпретация:** по темам видно, **какие направления бизнеса** чаще всего обсуждают клиенты в проанализированных отзываах. Рост доли `delivery` может сигнализировать о нагрузке на логистику; рост `support` — о нагрузке на линию поддержки.

---

## Пример 2: аналитика по оценкам (рейтингам)

**Агрегат по полю `rating` в `reviews`:**

- **Средняя оценка** считается **только по строкам**, где `rating` не `NULL`.
- Если ни у одного отзыва оценка не заполнена, в пользовательском отчёте отображается формулировка вида: средняя оценка **не указана** (нет отзывов с баллом).

**Учебный пример:** средняя оценка **3,4** по 10 отзывам с указанным баллом — **умеренный** уровень удовлетворённости; имеет смысл смотреть пересечение с темами (`negative` + `delivery` и т.д.).

**Интерпретация:** средний балл **не заменяет** разбор текстов; он даёт **быстрый числовой ориентир** при наличии данных.

---

## Пример 3: аналитика по продуктам

**Фрагмент сводки по `product_name`:**

| Товар (как в БД) | Число отзывов | Средний балл (где есть оценки) |
|------------------|---------------|--------------------------------|
| Блендер «Вихрь-300» | 3 | 2,7 |
| Кофемашина «Арома 9» | 2 | 5,0 |

**Интерпретация:** видно, по **каким позициям** больше всего обратной связи и как **оценки** соотносятся с объёмом отзывов. Для решений всё равно нужен разбор текстов и тем из `review_analysis`.

## Структура текстового отчёта (`format_report_ru`)

Типовой порядок блоков (компактная версия для Telegram):

1. **Заголовок** и общее число отзывов в выборке.
2. **По этапам обработки** — распределение по `status` (новый / обработан / ошибка).
3. **По тональности** — распределение по `sentiment` из анализа.
4. **По темам** — распределение по `topic` из анализа.
5. **Средняя оценка** — по полю `rating`, с пояснением, если данных нет.
6. **«Сложные» темы** — темы, где в анализе преобладают `negative` и `mixed`.
7. **По товарам** — количество отзывов и средний балл по названию товара (если для группы нет оценок — в интерфейсе показывается **«не указана»** для среднего по группе).

## Связь с кодом

| Компонент | Файл |
|-----------|------|
| Агрегаты SQL | `services/report_service.py` |
| Русские подписи | `services/localization_service.py` |
| Команда бота | `bot/handlers.py` → `/report` |

Представление **`review_full_view`** в SQLite (создаётся при `init_database`) удобно для выборок «отзыв + анализ» в одной строке; агрегаты отчёта считаются отдельными запросами в `report_service.py`.
